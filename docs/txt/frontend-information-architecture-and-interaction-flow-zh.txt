这份文档补两块内容：

1. 前端页面信息架构
2. 交互流设计

这是一份前端产品设计参考，不是当前 `apps/web` 实现的逐项说明。当前仓库已经有 workbench demo、事件归并视图模型和 block 编辑相关状态模型；本文讨论的是如果继续扩展成更完整的 AI 工作台，前端信息架构和交互流应该怎么设计。

目标是服务一个“多模型 + 交互式 AI 编程工作台”的重构版本。重点不是视觉稿，而是前端应该承载哪些信息、哪些交互节点必须存在、用户从开始到完成任务的路径应该怎么走。

====================
一、先定前端设计目标
====================

这个产品前端不是普通聊天页，也不是普通在线 IDE，而是一个：

“用户、AI 代理、运行时环境三方协作界面”

所以前端必须同时解决 4 件事：

1. 用户表达意图
2. 用户理解 AI 当前准备做什么
3. 用户在关键节点介入
4. 用户看到执行后的真实结果

如果只做好第 1 和第 4，产品就会退化成：
- 左边聊天
- 右边代码

这还不够。

真正要做的是：
- AI 的思路能被用户感知
- AI 的动作能被用户控制
- AI 的结果能被用户验证

另外，前端最好预留一个“交互式网页编辑接口”：
- 用户可以直接选中某个页面块
- 用自然语言描述这个块想怎么改
- 再补少量结构化属性，比如标题、语气、布局重点、按钮文案

这样后面细化网页时，不必每次都退回成整页级别的大改。

====================
二、整体信息架构
====================

推荐桌面端用三栏结构：

1. 左栏：导航与会话
2. 中栏：对话、计划、交互
3. 右栏：代码、终端、预览

如果展开成页面模块，可以是：

- TopBar
- LeftSidebar
- CenterWorkspace
- RightWorkbench
- BottomStatusBar

--------------------------------
1. TopBar
--------------------------------

应该放：
- 当前项目名
- 当前会话状态
- 当前模型配置入口
- 运行时类型标识
- 保存状态
- 分享 / 导出入口

作用：
- 给用户一个稳定的“当前上下文感”

--------------------------------
2. LeftSidebar
--------------------------------

应该放：
- 项目切换
- 会话历史
- 最近打开项目
- 收藏模板
- 设置入口

作用：
- 管理“你现在在哪个项目、哪段会话里”

不要把太多执行细节放左边，左边主要负责导航。

--------------------------------
3. CenterWorkspace
--------------------------------

这是产品的大脑区，建议拆成上下两层或者 tab：

- ChatPanel
- PlanPanel
- InteractionPanel
- WebEditorPanel
- ActionTimeline

作用：
- 告诉用户 AI 正在理解什么、计划什么、执行什么、还需要用户决定什么
- 在需要细化网页局部区域时，允许直接发起 block 级别的编辑请求

--------------------------------
4. RightWorkbench
--------------------------------

建议包含：
- FileTree
- EditorPanel
- TerminalPanel
- PreviewPanel
- DiffPanel

作用：
- 展示“真实项目状态”

这一栏是用户验证 AI 有没有真正完成任务的核心区域。

--------------------------------
5. BottomStatusBar
--------------------------------

可以放：
- 当前命令执行状态
- 端口状态
- 当前分支或版本标签
- 文件保存状态
- token / credit 使用提示

作用：
- 承载持续变化但不需要抢主视觉的信息

====================
三、页面模块详细建议
====================

--------------------------------
1. ChatPanel
--------------------------------

核心职责：
- 接收用户自然语言输入
- 展示用户消息和 assistant 输出
- 支持补充上下文

建议功能：
- 普通输入框
- 上传图片/设计稿/参考文件
- 引用当前文件
- 快捷插入“只修改当前文件”“不要执行命令”“先给计划”
- 中止生成按钮

注意：
- ChatPanel 不应该成为全部交互的唯一入口
- 计划确认、风险确认、选择题，不应该都混进聊天消息里

--------------------------------
2. PlanPanel
--------------------------------

这是重构时最应该补强的区域。

核心职责：
- 用结构化步骤展示 AI 当前打算怎么做
- 明示哪些步骤需要用户确认

建议展示：
- 计划标题
- 步骤列表
- 每步状态
- 是否需要确认
- 当前执行到哪一步

建议支持：
- 用户手动勾选跳过某一步
- 用户修改计划顺序
- 用户要求 AI 重新规划

如果没有 PlanPanel，用户只能被动等 AI 开始改代码，体验会更像黑箱。

--------------------------------
3. InteractionPanel
--------------------------------

核心职责：
- 集中承载需要用户介入的交互

建议类型：
- ChoiceCard
- ConfirmCard
- InputCard
- ConflictCard

典型场景：
- 选技术栈
- 是否允许安装依赖
- 是否覆盖用户手改文件
- 命令有风险是否继续
- 缺少信息时请用户补充

这个面板比把问题塞回聊天区更清晰，因为它让“待处理事项”显式可见。

--------------------------------
4. ActionTimeline
--------------------------------

核心职责：
- 展示 AI 执行了哪些动作
- 告诉用户当前执行状态

建议展示：
- 创建文件
- 修改文件
- 运行命令
- 启动预览
- 用户确认点
- 失败点

建议支持：
- 展开 stdout/stderr
- 点击跳到对应文件
- 点击查看 patch

这部分本质上是“把 agent 的执行过程可视化”。

--------------------------------
5. DiffPanel
--------------------------------

核心职责：
- 展示本轮 AI 要改什么
- 让用户决定是否接受

建议两种模式：
- 自动执行模式：默认直接执行，用户可回看 diff
- 审批模式：高风险变更先展示 diff，再确认应用

建议支持：
- 按文件查看 diff
- 只应用部分变更
- 忽略某个文件变更

如果你想做“交互式设计”，DiffPanel 是必须有的。

--------------------------------
6. FileTree
--------------------------------

核心职责：
- 展示项目文件结构
- 标出新文件、修改文件、未保存文件、冲突文件

建议状态标记：
- new
- modified
- unsaved
- conflict
- generated

--------------------------------
7. EditorPanel
--------------------------------

核心职责：
- 展示和编辑当前文件

--------------------------------
8. WebEditorPanel
--------------------------------

核心职责：
- 面向“网页内容和布局块”的轻量交互式编辑入口
- 让用户不一定非要直接写代码，也能细化某个页面局部

建议最小接口：
- `selection`
  指向当前页面块，比如 hero、feature-grid、footer
- `intent`
  用户用自然语言描述想改什么
- `patchStrategy`
  replace / append / refine
- `properties`
  少量结构化字段，比如标题、描述、按钮文案、视觉重点、布局方向

建议交互流程：
1. 用户在预览区或页面结构面板里选中某个 block
2. 前端把 block id、selector、当前摘要带入编辑面板
3. 用户填写 intent 和 properties
4. 系统把它转成更精确的 patch 请求，再交给 coder 或页面编辑能力

这个面板的价值在于：
- 把“改网页”从整文件编辑，下降到“改页面块”
- 让后续做可视化细化、设计稿对齐、营销页微调更自然
- 也更适合接未来的组件级 diff 和局部预览

建议支持：
- AI 高亮本轮改动
- 用户手动编辑
- 保存/还原
- 一键把当前文件发给 AI 作为上下文

重点：
- 编辑器必须支持“人和 AI 共同编辑”

--------------------------------
8. TerminalPanel
--------------------------------

核心职责：
- 展示命令输出
- 支持必要的人工干预

建议支持：
- 多 tab
- stdout / stderr 区分
- 复制命令
- 杀掉进程

--------------------------------
9. PreviewPanel
--------------------------------

核心职责：
- 展示当前应用运行结果

建议支持：
- 多端口切换
- 刷新
- 在新窗口打开
- 同步当前页面路径

如果是前端场景，PreviewPanel 是用户感知价值最直接的区域。

====================
四、核心用户流程
====================

建议至少明确 6 条主流程。

--------------------------------
流程 1：新建项目
--------------------------------

步骤：
1. 用户进入首页
2. 用户输入需求
3. AI 返回结构化计划
4. 如果有技术栈歧义，要求用户选择
5. 用户确认后，系统开始执行
6. 文件树、终端、预览逐步出现

这个流程的关键点：
- 先计划，再执行
- 关键选择不要埋进长段聊天文字里

--------------------------------
流程 2：在现有项目上继续迭代
--------------------------------

步骤：
1. 用户打开旧项目
2. 页面恢复会话、计划、文件树和预览
3. 用户提新需求
4. 系统读取当前文件状态和用户改动 diff
5. AI 基于最新状态生成更新计划
6. 系统执行

这个流程的关键点：
- AI 必须感知当前真实项目状态，而不是只看历史聊天

--------------------------------
流程 3：遇到歧义时请求用户选择
--------------------------------

步骤：
1. AI 无法确定技术方案
2. 触发 `interaction.choice`
3. InteractionPanel 显示选项卡
4. 用户选择一种方案
5. 系统继续执行

典型例子：
- React 还是 Next.js
- TypeScript 还是 JavaScript
- Tailwind 还是普通 CSS

--------------------------------
流程 4：遇到高风险操作时请求确认
--------------------------------

步骤：
1. AI 要执行安装、删除、大规模改动
2. 触发 `interaction.confirm`
3. 弹层或卡片展示风险摘要
4. 用户确认或拒绝
5. 系统继续或中止

典型例子：
- 删除目录
- 覆盖多个核心文件
- 执行潜在危险命令

--------------------------------
流程 5：用户与 AI 共同编辑文件
--------------------------------

步骤：
1. AI 改完一轮代码
2. 用户在 EditorPanel 手动编辑
3. 文件进入 modified/unsaved 状态
4. 用户再次提需求
5. 系统先保存并生成 diff
6. AI 基于最新内容继续修改

这个流程的关键点：
- 用户手动编辑不能被系统忽略
- 如果 AI 会覆盖用户修改，必须告知用户

--------------------------------
流程 6：错误恢复
--------------------------------

步骤：
1. 命令失败或预览启动失败
2. ActionTimeline 标出失败动作
3. ChatPanel 自动生成错误上下文摘要
4. 用户可选择：
   - 让 AI 自动修复
   - 查看日志
   - 回滚本轮改动

这个流程的关键点：
- 错误不能只 toast 一下就结束
- 要给出下一步路径

====================
五、交互设计最该强调什么
====================

重构时最重要的不是把界面做得像 IDE，而是把“控制权分配”做清楚。

建议控制权原则：

1. 普通低风险操作
默认自动执行

2. 中风险操作
允许用户在设置里选择“自动执行”或“先审查”

3. 高风险操作
必须确认

4. 有歧义的问题
必须让用户选

5. 与用户本地修改冲突
必须提示

这比花很多时间做炫技动画更重要。

====================
六、移动端和小屏策略
====================

这个产品天然偏桌面，但仍要有移动端策略。

建议：

1. 手机端默认单栏
- Chat / Plan / Code / Preview 用底部 tab 切换

2. 平板端双栏
- 左中合并
- 右侧 workbench 可抽屉化

3. 小屏上优先保留：
- ChatPanel
- PlanPanel
- PreviewPanel

文件树和终端可以放二级层。

====================
七、推荐页面层级
====================

推荐信息层级顺序：

第一层：当前目标和当前状态
- 你现在在哪个项目
- AI 正在做什么

第二层：用户当前需要处理什么
- 是否需要选择
- 是否需要确认

第三层：执行过程
- 哪些动作已完成
- 哪一步失败

第四层：结果验证
- 文件改了什么
- 页面跑起来没有

不要把终端日志放到最显眼位置，也不要让聊天消息淹没计划和审批信息。

====================
八、前端状态建议
====================

前端建议分 4 类状态：

1. 会话状态
- messages
- plan
- actions
- pendingInteraction

2. 工作台状态
- selectedFile
- openTabs
- terminalState
- previewState

3. UI 状态
- panel layout
- modal open/close
- theme

4. 本地编辑状态
- unsavedFiles
- draft inputs
- staged diff review

不要把所有状态都塞到一个 store 里。
建议：
- 会话状态走事件流 reducer
- 工作台状态走轻量 store
- UI 状态独立管理

====================
九、第一版前端最小页面集合
====================

MVP 前端建议只做下面这些页面：

1. 首页 / 工作台页
- 聊天
- 计划
- 交互确认
- 文件树
- 编辑器
- 终端
- 预览

2. 项目列表页
- 最近项目
- 新建项目

3. 设置页
- 模型偏好
- 自动执行偏好
- 风险确认策略

如果再加更多页面，前期容易分散精力。

====================
十、最推荐的交互路线
====================

如果现在就要做一版最稳的交互流，我建议：

1. 用户在 ChatPanel 提需求
2. PlanPanel 先展示 AI 计划
3. 如果有歧义，InteractionPanel 收集选择
4. 用户确认后，ActionTimeline 开始滚动执行
5. 右侧 Workbench 同步显示文件、终端、预览
6. 如果有高风险改动，弹确认
7. 如果失败，停在失败节点并给恢复路径
8. 用户修改文件后，再发起下一轮

一句话总结：

前端信息架构要服务“理解、控制、验证”三件事。聊天只是入口，真正让产品成立的是计划可见、风险可控、结果可验证。
