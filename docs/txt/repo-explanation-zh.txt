这个 repo 本质上是一个跑在 Cloudflare Pages + Remix + React 上的前端应用，但它的核心能力不只是聊天 UI，而是把大模型输出转成结构化“动作”，再在浏览器里的 WebContainer 中执行这些动作，最终让用户直接在页面里看到代码、终端和预览。

整体架构

这个仓库不是 monorepo，而是单应用。构建和开发入口在 /Users/bangxiao/Documents/bolt.new/package.json 和 /Users/bangxiao/Documents/bolt.new/vite.config.ts。前端框架是 Remix，SSR 入口是 /Users/bangxiao/Documents/bolt.new/app/entry.server.tsx，客户端 hydration 入口是 /Users/bangxiao/Documents/bolt.new/app/entry.client.tsx，全局壳子在 /Users/bangxiao/Documents/bolt.new/app/root.tsx。部署时通过 Cloudflare Pages Function 的兜底处理器 /Users/bangxiao/Documents/bolt.new/functions/[[path]].ts 把所有请求交给 Remix server build。

页面主路由很简单：首页 /Users/bangxiao/Documents/bolt.new/app/routes/_index.tsx 渲染 Header + Chat，聊天历史页 /Users/bangxiao/Documents/bolt.new/app/routes/chat.$id.tsx 其实复用了首页组件，只是 loader 会把 id 带下来，用来恢复历史会话。

它怎么“运作”

这套产品的主链路是：

1. 用户在聊天框输入需求。
2. 前端用 useChat 调 /api/chat。
3. 服务端把消息发给 Anthropic 模型，并强制模型遵守自定义协议输出。
4. 前端一边流式显示文本，一边解析模型回复里的 <boltArtifact> / <boltAction>。
5. 解析到的 file 动作会写入 WebContainer 文件系统，shell 动作会在浏览器内 shell 里执行。
6. 文件树、编辑器、终端、预览面板都订阅同一个工作台状态，所以会实时更新。

这条链路的前端入口在 /Users/bangxiao/Documents/bolt.new/app/components/chat/Chat.client.tsx。它做了几件关键事：
- 用 useChat({ api: '/api/chat' }) 发起流式对话。
- 在发送消息前调用 workbenchStore.saveAllFiles()，确保用户在编辑器里未保存的修改先落盘。
- 如果用户改过文件，会把 diff 包进用户消息里一起发给模型，让模型知道当前项目状态。
- 收到模型流式输出后，交给 useMessageParser() 解析成可执行动作。

服务端 API 在 /Users/bangxiao/Documents/bolt.new/app/routes/api.chat.ts。它不直接写业务逻辑，而是调用 /Users/bangxiao/Documents/bolt.new/app/lib/.server/llm/stream-text.ts。这里会：
- 读取 API Key: /Users/bangxiao/Documents/bolt.new/app/lib/.server/llm/api-key.ts
- 固定使用 Anthropic 模型: /Users/bangxiao/Documents/bolt.new/app/lib/.server/llm/model.ts
- 注入非常长的 system prompt: /Users/bangxiao/Documents/bolt.new/app/lib/.server/llm/prompts.ts

这个 system prompt 是整个 repo 的灵魂。它告诉模型：
- 运行环境是 WebContainer，不是完整 Linux。
- 不能跑原生二进制，不能用 pip，不能用 git。
- 必须把回复包装成 <boltArtifact> 和 <boltAction>。
- boltAction 只有两类：file 和 shell。
- 必须输出完整文件内容，且动作顺序要正确。

也就是说，这个系统不是“自由聊天”，而是“让模型按 DSL 协议生成可执行计划”。

前端如何把模型回复变成代码和命令

解析器在 /Users/bangxiao/Documents/bolt.new/app/lib/runtime/message-parser.ts。它是一个流式 parser，会逐字符扫描 assistant 输出：
- 遇到 <boltArtifact ...> 就创建一个工作单元。
- 遇到 <boltAction type="file" filePath="..."> 就收集文件内容。
- 遇到 <boltAction type="shell"> 就收集命令内容。
- 遇到关闭标签时，触发回调。

这些回调在 /Users/bangxiao/Documents/bolt.new/app/lib/hooks/useMessageParser.ts 里接到 workbenchStore：
- onArtifactOpen：打开 workbench，登记一次任务。
- onActionOpen/onActionClose：把动作加入当前任务。
- onActionClose：真正执行动作。

真正执行动作的是 /Users/bangxiao/Documents/bolt.new/app/lib/runtime/action-runner.ts。这里做了两件事：
- file 动作：在 WebContainer 里 mkdir + writeFile
- shell 动作：在 WebContainer 里 spawn('jsh', ['-c', command])

而且它用 #currentExecutionPromise 串行化执行，保证动作按模型给出的顺序跑，不会乱序。

补两点实现上的关键理解

第一，这个项目的“智能”主要不在 React 组件，而在那份 system prompt 和前端 parser 协议。模型只要守协议，前端就能把回复自动执行成项目文件和命令。

第二，它不是传统 SaaS 代码生成器那种“服务端跑容器”，而是把执行环境放进浏览器。也就是：
- 服务端只负责“调用模型并流式转发”
- 真正的文件系统、终端、预览、运行命令都在用户浏览器里的 WebContainer 中完成

所以它的核心价值可以概括成一句话：

这个 repo 是一个“基于协议驱动的浏览器内 AI 编程 IDE”。
Remix 负责页面和 API，Anthropic 负责生成结构化动作，Message Parser 负责把回复转成事件，WorkbenchStore 负责统一状态，WebContainer 负责真正执行项目并产出预览。

从一次用户发消息开始的完整调用链

1. 页面先启动

用户打开首页后，请求先到 Cloudflare Pages 的 /Users/bangxiao/Documents/bolt.new/functions/[[path]].ts，它把请求交给 Remix。
Remix SSR 入口是 /Users/bangxiao/Documents/bolt.new/app/entry.server.tsx，客户端接管在 /Users/bangxiao/Documents/bolt.new/app/entry.client.tsx。
真正首页组件是 /Users/bangxiao/Documents/bolt.new/app/routes/_index.tsx，里面渲染 Header 和 Chat。

2. Chat 组件初始化

/Users/bangxiao/Documents/bolt.new/app/components/chat/Chat.client.tsx 是主入口。它做三件事：
- 用 useChatHistory() 恢复 IndexedDB 里的历史消息
- 用 useChat({ api: '/api/chat' }) 建立和后端聊天 API 的流式连接
- 用 useMessageParser() 准备解析 assistant 的结构化输出

如果当前 URL 是 /chat/:id，那 /Users/bangxiao/Documents/bolt.new/app/lib/persistence/useChatHistory.ts 会先从 IndexedDB 取出历史 messages，作为 useChat 的 initialMessages。

3. 用户点击发送

用户在输入框里按回车，最终会调用 sendMessage()，位置还是 /Users/bangxiao/Documents/bolt.new/app/components/chat/Chat.client.tsx。

这里的顺序很关键：

先 await workbenchStore.saveAllFiles()
意思是：如果用户刚在右侧编辑器里改了文件但还没保存，先全部写回 WebContainer 文件系统。

然后取 workbenchStore.getFileModifcations()
这一步会从 /Users/bangxiao/Documents/bolt.new/app/lib/stores/files.ts 里拿到“自上次发消息以来，哪些文件被用户改了”。

如果有改动，就把 diff 包成一段特殊文本插到用户消息前面，再 append({ role: 'user', content: ... })。
这样模型拿到的不是纯自然语言，而是：
- 用户这次的新需求
- 当前工程里用户已经改过的文件差异

这就是它能“持续接着改项目”的关键。

4. 请求进入 /api/chat

前端 useChat 会调用 /Users/bangxiao/Documents/bolt.new/app/routes/api.chat.ts。

这里会：
- 读取请求体里的 messages
- 创建 SwitchableStream
- 调用 /Users/bangxiao/Documents/bolt.new/app/lib/.server/llm/stream-text.ts

stream-text.ts 会把这些消息发给 Anthropic，并自动附带 system prompt，system prompt 在 /Users/bangxiao/Documents/bolt.new/app/lib/.server/llm/prompts.ts。

这份 prompt 强制模型按下面格式输出：

<boltArtifact id="..." title="...">
  <boltAction type="file" filePath="...">...</boltAction>
  <boltAction type="shell">...</boltAction>
</boltArtifact>

也就是说，后端并不理解“如何改代码”，它只负责把上下文喂给模型，并把模型输出流式返回。

5. assistant 输出开始流回来

前端 useChat 一边收到 token，一边把 assistant 消息更新到 messages 里。
接着 Chat.client.tsx 里的 useEffect 会触发：
- parseMessages(messages, isLoading)
- storeMessageHistory(messages)

前者做“执行”，后者做“持久化”。

6. 流式解析 assistant 回复

parseMessages() 来自 /Users/bangxiao/Documents/bolt.new/app/lib/hooks/useMessageParser.ts，底层是 /Users/bangxiao/Documents/bolt.new/app/lib/runtime/message-parser.ts。

这个 parser 是增量式的。assistant 回复哪怕只回来一半，它也会从上次解析到的位置继续扫。

它会识别四个关键事件：
- onArtifactOpen
- onArtifactClose
- onActionOpen
- onActionClose

例如：
- 读到 <boltArtifact ...> 时，触发 onArtifactOpen
- 读到 <boltAction type="file"...> 时，开始收集文件内容
- 读到 </boltAction> 时，说明这个动作完整了，触发 onActionClose

7. 解析事件进入 Workbench

这些回调都接到 /Users/bangxiao/Documents/bolt.new/app/lib/stores/workbench.ts。

具体是：
- onArtifactOpen -> workbenchStore.addArtifact(...)
- onActionOpen / onActionClose -> workbenchStore.addAction(...)
- onActionClose -> workbenchStore.runAction(...)

这里有个重要设计：
每个 assistant 消息会对应一个 ActionRunner，定义在 /Users/bangxiao/Documents/bolt.new/app/lib/runtime/action-runner.ts。

ActionRunner 会把动作排队串行执行，避免同时写文件、同时跑命令导致状态混乱。

8. 真正执行 file / shell

ActionRunner 分两种：

file 动作：
- 确保目录存在
- 调 webcontainer.fs.writeFile(...)

shell 动作：
- 调 webcontainer.spawn('jsh', ['-c', command])
- 把输出流接到控制台

也就是说，模型给出的命令和文件，不是在服务器上执行，而是在浏览器里的 WebContainer 执行。

WebContainer 初始化在 /Users/bangxiao/Documents/bolt.new/app/lib/webcontainer/index.ts，工作目录是 /home/project，定义在 /Users/bangxiao/Documents/bolt.new/app/utils/constants.ts。

这里是整个 repo 最漂亮的一段闭环。

文件变化：
/Users/bangxiao/Documents/bolt.new/app/lib/stores/files.ts 会监听 WebContainer 文件系统事件。
一旦模型写了新文件或改了旧文件，files store 立刻更新。

于是：

所以用户看到的是：AI 一边说，一边写文件，一边跑命令，一边右侧项目长出来。

/Users/bangxiao/Documents/bolt.new/app/components/chat/Markdown.tsx 会把 parser 注入的 __boltArtifact__ 占位节点替换成 /Users/bangxiao/Documents/bolt.new/app/components/chat/Artifact.tsx。

Store 和状态协作

总控：workbenchStore

核心是 /Users/bangxiao/Documents/bolt.new/app/lib/stores/workbench.ts。
它不是单纯存一个对象，而是把几个子系统组合起来：

- PreviewsStore：管理预览地址
- FilesStore：管理文件树和文件内容
- EditorStore：管理当前编辑中的文档
- TerminalStore：管理终端实例和显隐
- ActionRunner：按顺序执行 AI 生成的动作

workbenchStore 对外暴露的是整个右侧 IDE 面板需要的统一接口，比如：
- 当前选中文件
- 当前编辑文档
- 文件列表
- 未保存文件
- 当前视图是 code 还是 preview
- 是否显示 terminal
- 执行 AI action
- 保存文件 / 保存全部文件

所以组件基本不直接碰 WebContainer，而是通过 workbenchStore 间接操作。

1. FilesStore：文件系统真相源

/Users/bangxiao/Documents/bolt.new/app/lib/stores/files.ts 是“项目文件状态”的真相源。

它的职责：
- 监听 WebContainer 文件系统变化
- 维护 files: MapStore<FileMap>
- 区分文件和文件夹
- 判断二进制文件，避免编辑器误显示
- 保存用户修改
- 记录“自上次发消息以来哪些文件被用户改过”

这里最关键的是两个状态：
- files
- #modifiedFiles

files 表示当前 WebContainer 里真实存在的项目文件。
#modifiedFiles 表示用户改过哪些文件，以及原始内容是什么。这个状态专门给下一次发消息时生成 diff 用。

所以 FilesStore 既服务 UI，也服务 prompt 上下文。

2. EditorStore：编辑器当前打开了什么

EditorStore 的代码没单独展开，但从 /Users/bangxiao/Documents/bolt.new/app/lib/stores/workbench.ts 和 /Users/bangxiao/Documents/bolt.new/app/components/workbench/EditorPanel.tsx 的调用关系能看出来，它负责：

- 当前选中的文件路径
- 当前文档内容
- 滚动位置
- 文档和文件系统内容之间的映射

它和 FilesStore 的关系是：
- FilesStore 管“磁盘”
- EditorStore 管“编辑会话中的文档状态”

所以用户在编辑器里敲字时，并不是每个字符都直接写回 WebContainer；而是先更新 EditorStore，点击保存或发送消息前，再由 workbenchStore.saveFile/saveAllFiles() 写回 FilesStore/WebContainer。

3. TerminalStore：终端连接层

/Users/bangxiao/Documents/bolt.new/app/lib/stores/terminal.ts 很纯粹，它不负责 shell 逻辑本身，只负责：

- showTerminal 显隐状态
- 维护已挂接的 terminal 实例
- 在 WebContainer 里创建 shell 进程
- 终端 resize 时同步给进程

真正创建 shell 的细节在 /Users/bangxiao/Documents/bolt.new/app/utils/shell.ts。
这里会 spawn('/bin/jsh', ['--osc'])，等 shell 进入 interactive 模式后，把用户输入和终端输出对接起来。

所以 TerminalStore 本质上是：
xterm UI 和 WebContainer shell 之间的桥。

4. PreviewsStore：预览端口目录

/Users/bangxiao/Documents/bolt.new/app/lib/stores/previews.ts 专门监听 WebContainer 的 port 事件。

职责只有三个：
- 记录当前有哪些可访问端口
- 记录每个端口是否 ready
- 给出对应 baseUrl

这意味着它根本不关心项目是 React、Vite、Next 还是别的。
只要容器里某个服务启动并暴露端口，它就记下来。

/Users/bangxiao/Documents/bolt.new/app/components/workbench/Preview.tsx 再从这个 store 里拿 URL，塞进 iframe。

所以预览系统的设计很干净：
- 后端不参与
- 模型也不直接控制 iframe
- 纯靠 WebContainer 的端口事件驱动

5. chatStore：聊天 UI 的轻量状态

/Users/bangxiao/Documents/bolt.new/app/lib/stores/chat.ts 很简单，只有：

- aborted
- showChat

它主要服务聊天区 UI，比如：

- 某些动画/布局切换

也就是说，chatStore 不是业务核心，只是聊天面板的局部 UI 状态。

6. themeStore：主题状态

/Users/bangxiao/Documents/bolt.new/app/root.tsx 和编辑器、终端都会订阅主题状态。
它主要影响：
- HTML data-theme
- CodeMirror 主题
- Terminal 样式
- 全局 CSS 变量

这是典型的全局展示状态，不参与业务链路。

7. ActionRunner：不是 store，但它像状态机

/Users/bangxiao/Documents/bolt.new/app/lib/runtime/action-runner.ts 严格说不是 UI store，但它维护了一份 actions: MapStore<Record<string, ActionState>>，所以在系统里扮演了“执行态 store”的角色。

每个 action 有：
- pending
- running
- complete
- aborted
- failed

聊天区里的 /Users/bangxiao/Documents/bolt.new/app/components/chat/Artifact.tsx 就是订阅这个状态来显示每一步执行进度。

所以如果说 FilesStore 保存的是“项目状态”，那 ActionRunner.actions 保存的就是“执行状态”。

这些 store 怎么协作

一次典型协作链路是这样的：

1. 用户在编辑器改文件
EditorStore 更新当前文档内容
workbenchStore 根据原始文件内容，更新 unsavedFiles

2. 用户点击发送
workbenchStore.saveAllFiles()
把 EditorStore 内容写回 FilesStore/WebContainer

3. 前端发消息前
FilesStore.getFileModifications() 生成 diff
把 diff 注入 prompt 发给模型

4. 模型流式回复
useMessageParser() 解析出 action
workbenchStore.addAction/runAction()

5. ActionRunner 执行
如果是 file，写 WebContainer 文件系统
如果是 shell，在 WebContainer 里跑命令

6. WebContainer 有变化
FilesStore 监听到文件变动
PreviewsStore 监听到端口变动
TerminalStore 把 shell 输出写进终端

7. UI 自动刷新
文件树、编辑器、执行卡片、预览 iframe 全都因为订阅 store 而更新

最值得注意的设计点

这个 repo 的状态设计有一个很明确的分层：

- WebContainer 是运行时真相
- FilesStore / PreviewsStore / TerminalStore 是运行时同步层
- WorkbenchStore 是编排层
- React 组件只是订阅和触发操作

这意味着它不是“组件里塞一堆状态”的写法，而是比较典型的：
运行环境驱动状态，状态驱动 UI。

一句话记忆

如果你只记一件事，可以记这个：

- FilesStore 管项目文件
- EditorStore 管当前编辑态
- TerminalStore 管 shell
- PreviewsStore 管端口预览
- ActionRunner 管 AI 动作执行状态
- WorkbenchStore 把上面这些全部串起来

组件之间怎么连接这些 store

1. 最外层页面怎么组装

首页 /Users/bangxiao/Documents/bolt.new/app/routes/_index.tsx 很薄，只负责渲染：

这意味着任务卡片不是静态文本，而是直接绑定执行状态：

这里的交互方向是双向的：

所以 EditorPanel 是典型的受控面板，状态真相不在组件里，而在 workbenchStore 背后的 stores。

所以终端组件本身只负责显示和输入采集，真正的 shell 进程生命周期在 store 层。
