这份文档聚焦两件事：

1. 模型路由策略
2. Prompt 分层设计

这是一份面向下一阶段演进的设计文档，不是当前仓库实现状态说明。当前代码里 `packages/model-gateway` 仍然是 stub；这里讨论的是后续接入真实模型时推荐采用的路由与 prompt 分层方式。

目标是服务一个“多模型 + 交互式 AI 编程工作台”的重构版本。重点不是教你怎么写一句 prompt，而是从系统设计角度说明：
- 为什么要多模型分工
- 哪类任务该交给哪类模型
- prompt 不应该怎样堆在一起
- 怎么让系统既可控又容易持续迭代

====================
一、为什么一定要做模型路由
====================

如果你只是做一个普通聊天产品，用一个默认模型从头跑到尾，也许还能接受。

但如果你要做的是一个 AI 编程工作台，任务类型已经明显不止一种：

- 理解用户意图
- 生成结构化计划
- 生成或修改代码
- 审查高风险改动
- 总结会话
- 给历史会话命名
- 解释错误日志
- 与用户做选择题交互

这些任务的最优模型往往不是同一个。

如果始终用同一个大模型，你会遇到几个问题：

1. 成本高
很多轻量任务没必要用最贵、最强的模型。

2. 延迟高
标题、摘要、简单归纳完全没必要走重推理。

3. 行为不稳定
擅长规划的模型未必最适合长链路代码修改。

4. 难优化
所有问题都混在一个模型里，你很难定位到底是规划差、编码差，还是审查差。

所以重构时应该把“模型”从一个全局默认值，升级成“按任务路由的能力池”。

====================
二、模型路由的总体原则
====================

建议遵循 5 条原则：

1. 先按任务类型路由，不要按页面路由

正确思路：
- planning 任务给 planning 模型
- coding 任务给 coding 模型
- summary 任务给轻量模型

不要写成：
- 聊天页用 A 模型
- 编辑器页用 B 模型

因为一个页面里本来就可能同时发生多种任务。

2. 先做“能力分工”，再做“成本优化”

第一版要先追求行为清晰，不要一上来搞很复杂的成本路由。

先把角色分清：
- 谁负责规划
- 谁负责写代码
- 谁负责审查
- 谁负责摘要

稳定后再加：
- 根据 token 预算动态切换
- 根据任务复杂度降级

3. 不要让前端知道具体模型

前端只知道任务类型和事件，不应该感知：
- 这次是 GPT-5.4
- 那次是 GPT-5-Codex

模型选择应该在 `model-gateway` 或 `agent service` 内部完成。

4. 路由策略要可配置，不要写死

一定要把模型映射做成配置，例如：

```ts
planning: gpt-5.4
coding: gpt-5-codex
review: gpt-5.4
summary: gpt-5-mini
title: gpt-5-mini
```

以后你只改配置，不需要改业务代码。

5. 路由后仍然要统一协议

无论底层调用哪个模型，上层拿到的都应该是统一结构：
- plan
- action
- review result
- summary

而不是不同模型各返回一种奇怪格式。

====================
三、推荐模型角色划分
====================

这里不写死某一个供应商，只写建议角色。你当前目标是接入 GPT、Codex 系列，所以可以先按这组分工理解。

--------------------------------
1. Planner 模型
--------------------------------

职责：
- 理解用户真正目标
- 把模糊需求拆成结构化 plan
- 识别需要用户确认的点
- 决定下一步该调用哪个子能力

适合特征：
- 推理强
- 多轮上下文稳定
- 结构化输出稳定

推荐定位：
- 用旗舰推理模型，例如 GPT-5.4

Planner 不应该直接写大量代码。它更像项目经理。

--------------------------------
2. Coder 模型
--------------------------------

职责：
- 根据 plan 生成文件改动
- 编写 patch
- 生成命令
- 修改已有代码

适合特征：
- 代码编辑能力强
- 长上下文代码理解强
- 对工程结构敏感

推荐定位：
- 用 coding 特化模型，例如 GPT-5-Codex 或 GPT-5.3-Codex

Coder 不应该承担太多高层规划职责，否则输出容易一会儿像 PM，一会儿像程序员。

--------------------------------
3. Reviewer 模型
--------------------------------

职责：
- 审查高风险改动
- 检查是否覆盖用户本地修改
- 检查明显逻辑错误、缺失步骤、潜在回归

适合特征：
- 推理严谨
- 更保守
- 擅长比较“原状态”和“目标状态”

推荐定位：
- 用高质量推理模型，例如 GPT-5.4

Reviewer 的输出通常不是直接落文件，而是：
- 风险列表
- 建议确认项
- 审查意见

--------------------------------
4. Summarizer 模型
--------------------------------

职责：
- 压缩历史会话
- 总结项目状态
- 生成短标题
- 生成阶段性摘要

适合特征：
- 成本低
- 速度快
- 结构化归纳稳定

推荐定位：
- 用轻量模型，例如 GPT-5-mini

这个角色最适合降本。

--------------------------------
5. Interaction 模型
--------------------------------

职责：
- 把歧义问题转成用户容易回答的问题
- 把复杂风险解释成清晰确认文案
- 在用户回应后转成结构化输入

这个角色可选。
早期版本里，也可以由 Planner 兼任。

====================
四、推荐的模型路由配置
====================

建议做成可配置的 profile。

例如：

--------------------------------
Profile A：质量优先
--------------------------------

- planning -> gpt-5.4
- coding -> gpt-5-codex
- review -> gpt-5.4
- summary -> gpt-5-mini
- title -> gpt-5-mini

适用：
- 复杂项目
- 高风险修改
- 对质量要求高

--------------------------------
Profile B：速度优先
--------------------------------

- planning -> gpt-5-mini
- coding -> gpt-5.3-codex
- review -> gpt-5-mini
- summary -> gpt-5-mini
- title -> gpt-5-mini

适用：
- 原型搭建
- 简单页面生成
- 低成本模式

--------------------------------
Profile C：平衡模式
--------------------------------

- planning -> gpt-5.4
- coding -> gpt-5.3-codex
- review -> gpt-5-mini
- summary -> gpt-5-mini
- title -> gpt-5-mini

适用：
- 默认产品模式

====================
五、什么时候切模型
====================

建议按“任务阶段”切，而不是按“用户第几轮对话”切。

推荐阶段：

1. 用户提出需求
-> planner

2. 生成计划并识别确认点
-> planner

3. 用户确认后开始做代码改动
-> coder

4. 要执行高风险改动前
-> reviewer

5. 每轮结束后压缩上下文
-> summarizer

6. 给会话命名
-> summarizer

不要这样做：
- 第一轮都用 A
- 第二轮都用 B

这样没有任务语义。

====================
六、动态路由规则怎么设计
====================

稳定版本之后，可以加动态路由。建议从简单规则开始。

可以参考这些维度：

1. 任务复杂度

例如：
- “改个按钮颜色” -> 轻量 coder
- “重构认证流程” -> 强 coder + review

2. 风险等级

例如：
- 改一个展示页 -> 可直接 coder
- 改登录、支付、数据库迁移 -> 必须加 reviewer

3. 上下文规模

例如：
- 项目很大，上下文超长 -> 优先能处理长上下文的模型

4. 用户偏好

例如：
- 用户选择“质量优先”
- 用户选择“省钱优先”

5. 当前失败情况

例如：
- 轻量 coder 连续两次失败 -> 升级到更强 coder

建议一开始只做：
- 复杂度
- 风险
- 用户偏好

不要太早做黑盒自动优化，不然很难 debug。

====================
七、最重要的不是路由本身，而是输入输出边界
====================

很多系统路由做得乱，是因为每个模型看到的输入格式不统一、输出格式也不统一。

正确做法是先定义每个角色的 I/O。

--------------------------------
1. Planner 输入
--------------------------------

建议包含：
- 用户当前需求
- 当前项目摘要
- 最近关键文件状态
- 用户偏好
- 风险策略

Planner 输出：
- 结构化计划
- 是否需要交互
- 给 coder 的任务摘要

--------------------------------
2. Coder 输入
--------------------------------

建议包含：
- 通过审核的计划步骤
- 当前相关文件内容
- 用户最近修改 diff
- 运行时能力说明
- 输出 schema

Coder 输出：
- file actions
- command actions
- patch / file content

--------------------------------
3. Reviewer 输入
--------------------------------

建议包含：
- 原计划
- 待执行或已生成的动作
- 关键文件 diff
- 风险规则

Reviewer 输出：
- 风险列表
- 是否阻塞执行
- 是否需要确认

--------------------------------
4. Summarizer 输入
--------------------------------

建议包含：
- 最近消息
- 最新计划
- 动作结果
- 当前项目状态摘要

Summarizer 输出：
- 会话摘要
- 标题
- 压缩上下文

====================
八、Prompt 为什么必须分层
====================

如果把所有规则都堆到一个巨大 system prompt 里，短期能跑，长期一定失控。

常见问题：
- planner 看到了 coder 细节，开始直接写代码
- coder 看到了所有产品规则，开始输出冗长解释
- reviewer 也尝试规划整个项目
- 一改某段 prompt，就影响所有任务

所以 prompt 必须分层。

====================
九、推荐 Prompt 分层结构
====================

建议拆成 5 层：

1. 全局平台层
2. 角色层
3. 运行时层
4. 任务层
5. 输出协议层

--------------------------------
1. 全局平台层
--------------------------------

作用：
- 定义产品共同规则
- 定义安全边界
- 定义总体目标

例如内容：
- 这是一个 AI 编程工作台
- 需要尊重用户修改
- 高风险动作必须显式标出
- 不要伪造执行结果

这层所有角色共享。

--------------------------------
2. 角色层
--------------------------------

作用：
- 定义某个模型当前扮演的角色

例如：
- Planner：只负责规划，不直接写完整代码
- Coder：只负责生成和修改代码，不解释过多产品策略
- Reviewer：只负责风险审查，不直接重写整个方案

这一层是 prompt 分层里最重要的边界。

--------------------------------
3. 运行时层
--------------------------------

作用：
- 告诉模型当前运行环境能力和限制

例如：
- 当前是 browser runtime stub，或未来切换成 remote runtime
- 可用命令列表
- 不能跑原生二进制
- 文件系统根目录
- 是否允许联网

这层尤其适合 coder 和 reviewer。

--------------------------------
4. 任务层
--------------------------------

作用：
- 注入本轮具体任务

例如：
- 用户目标
- 当前计划步骤
- 当前相关文件
- 最近错误日志
- 用户选择结果

这一层变化最频繁。

--------------------------------
5. 输出协议层
--------------------------------

作用：
- 强制本轮返回格式

例如：
- Planner 必须返回 Plan schema
- Coder 必须返回 Action schema
- Reviewer 必须返回 Review schema
- Summarizer 必须返回 Summary schema

这一层最不应该写成自由文本描述，应该和 zod schema 对应。

====================
十、推荐 Prompt 组织方式
====================

建议不要把 prompt 全写在一个文件里。

推荐组织：

/infra/prompts
  /global
    platform.txt
    safety.txt

  /roles
    planner.txt
    coder.txt
    reviewer.txt
    summarizer.txt

  /runtime
    browser-runtime.txt
    remote-runtime.txt

  /tasks
    plan-task.ts
    code-task.ts
    review-task.ts
    summary-task.ts

  /schemas
    plan-schema.ts
    action-schema.ts
    review-schema.ts
    summary-schema.ts

这样做的好处：
- 改 planner 不会误伤 coder
- 改 runtime 约束不会误伤 summary
- 输出协议可以和 types 对齐

====================
十一、推荐给各角色的 Prompt 风格
====================

--------------------------------
1. Planner prompt 风格
--------------------------------

应该强调：
- 先理解，再拆解
- 计划要短、结构化、可执行
- 明确哪些地方有歧义
- 必要时请求用户确认

不应该强调：
- 输出完整代码
- 大段解释实现细节

--------------------------------
2. Coder prompt 风格
--------------------------------

应该强调：
- 只输出结构化动作
- 只改相关文件
- 优先最小必要变更
- 不覆盖用户最近修改
- 命令执行顺序要正确

不应该强调：
- 长篇解释
- 高层产品规划

--------------------------------
3. Reviewer prompt 风格
--------------------------------

应该强调：
- 找风险
- 找冲突
- 判断是否要阻塞
- 用简短清晰的语言说明原因

不应该强调：
- 顺手重写全部代码

--------------------------------
4. Summarizer prompt 风格
--------------------------------

应该强调：
- 压缩
- 结构化
- 保留关键决策
- 省 token

====================
十二、避免 prompt 污染
====================

多模型系统常见问题之一是 prompt 污染，也就是不该传给某个角色的信息传了进去。

例如：
- planner 看到了所有终端日志，反而变得啰嗦
- summarizer 看到了大量 runtime 约束，摘要质量变差
- coder 看到了历史所有计划和审查意见，开始过度修复

建议规则：

1. 每个角色只拿自己需要的上下文
2. 上下文长度超限时先做摘要
3. 错误日志只给和错误有关的角色
4. 文件内容只给相关文件，不要全项目一股脑塞进去

====================
十三、推荐的最小模型路由实现
====================

如果你现在只想做第一版，我建议先这样：

第一阶段：
- planner -> GPT-5.4
- coder -> GPT-5-Codex
- summary/title -> GPT-5-mini

review 暂时可以先由 planner 兼任。

当你有了更多高风险场景，再单独拆 reviewer。

====================
十四、推荐的失败回退策略
====================

模型路由不应该只考虑成功时，也要考虑失败时。

建议：

1. Planner 失败
- 回退到更简单的规划 prompt
- 或让系统直接请求用户补充信息

2. Coder 失败
- 重试一次
- 如果连续失败，升级到更强 coder

3. Reviewer 判定不清
- 直接转成用户确认

4. Summarizer 失败
- 退化成规则摘要，不阻塞主流程

====================
十五、最推荐的整体做法
====================

如果把上面全部压缩成一句工程建议，那就是：

1. 按任务角色切模型，不按页面切
2. 先定义角色 I/O，再定义路由
3. Prompt 一定做分层，不要把所有规则堆进一个大 system prompt
4. Planner、Coder、Summarizer 先拆开
5. 所有模型输出统一成 schema，前端和执行器只认 schema

一句话总结：

模型路由解决的是“谁来做哪类事”，Prompt 分层解决的是“它在这个角色里该怎么做”。前者负责分工，后者负责边界。两者一起做好，多模型系统才会稳。
